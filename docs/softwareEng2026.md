# ソフトウェア工学 まとめ
WBSとは、プロジェクト全体で行う作業を階層的に細かく分解し、整理するための手法である。ソフトウェア開発では、大きな作業を「要件定義」「設計」「実装」「テスト」「運用」などに分割し、さらにそれぞれを細かな作業へと分解していく。
WBSを作成することで、プロジェクト全体で必要となる作業を明確に把握でき、作業の抜け漏れや重複を防ぐことができる。また、各作業の担当者や開始日・終了日を決めやすくなり、進捗状況の管理もしやすくなる。
ソフトウェア開発では、一つのシステムを複数人で開発することが多いため、WBSによって作業内容を共有することで、チーム全体が自分の担当やプロジェクト全体の流れを理解しやすくなる。また、作業量を見積もる際の基準にもなり、開発期間や必要な人員を予測しやすくなるという利点がある。
さらに、WBSはガントチャートなどのスケジュール管理ツールと組み合わせて利用されることが多く、各作業の進捗状況や遅れを可視化できる。問題が発生した場合でも、どの作業に影響があるのかを把握しやすくなり、早期に対応できる。
WBSは単に作業を細かく分けるだけではなく、計画的かつ効率的にソフトウェア開発を進めるための基礎となる重要な手法であり、品質向上や納期の遵守、円滑なチーム開発を実現するために欠かせないものである。
ソフトウェア開発では、複数人で効率よく開発を進めるために、Gitを利用したバージョン管理やGitHubを用いた共同開発が重要であることを学んだ。Gitではブランチを利用して機能ごとに開発を進め、変更内容をコミットとして記録することで、開発履歴を管理できる。また、ブランチを切り替えたりマージしたりすることで、複数人が同時に作業しても互いの変更が競合しにくく、安全に開発を進められることを理解した。
GitHubでは、GitHub Flowという開発手法を学んだ。まずmainブランチから新しいブランチを作成し、そのブランチ上で作業を行う。変更が完了したらPushしてPull Requestを作成し、レビューを受けた後にmainブランチへマージする。この流れによって変更履歴を管理しやすくなり、コードレビューを通して品質を向上させながら開発を進められる。また、IssueやProjectsを利用することで、課題や進捗を整理し、チーム全体で情報を共有しやすくなる。
CIは、ソースコードの変更を共有リポジトリへ統合するたびに、自動でビルドやテストを実行し、不具合を早期に発見する仕組みである。一方、CDは、テストを通過したプログラムを自動的にデプロイし、利用者へ迅速に提供するための仕組みである。GitHub Actionsでは、YAMLファイルを用いてワークフローを定義し、PushやPull Requestをきっかけに自動で処理を実行できる。
GitHub PagesとJekyllを利用することで、GitHub上のリポジトリをそのままWebページとして公開できる。Markdown形式で作成した文章を自動的にHTMLへ変換できるため、専門的なWeb制作の知識がなくても見やすいWebサイトを作成できる。さらに、GitHub Actionsと組み合わせることで、リポジトリを更新すると自動的にWebページも更新されるため、常に最新の内容を公開できる。
今回の講義を通して、Gitによるバージョン管理からGitHubを利用した共同開発、CI/CDによる自動化、GitHub PagesによるWeb公開まで、実際のソフトウェア開発で利用されている一連の開発手法を学ぶことができた。これらの技術を組み合わせることで、開発効率の向上だけでなく、ソフトウェアの品質維持やチームでの円滑な共同作業につながることを理解した。今後の開発演習や実践的なシステム開発にも活用していきたいと感じた。
